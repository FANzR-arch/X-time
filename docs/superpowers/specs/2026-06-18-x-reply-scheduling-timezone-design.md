# X 回复排期与目标时区设计

日期：2026-06-18

> 本设计中的“X 原生回复排期”前提经能力核对后确认不成立。最终实现由扩展使用 `chrome.alarms` 到点发送，详见 `2026-06-18-x-extension-timed-replies-amendment.md`；修正文件覆盖本文冲突部分。

## 目标

在现有原创帖排期插件中增加独立的批量回复排期入口。用户导入多组“目标帖子链接 + 纯文字回复 + 可选指定时间”后，插件逐条打开目标帖子、填写回复、调用 X 原生排期并确认。原创帖和回复共享一个可选择的目标时区；排期规则按目标时区解释，再转换为浏览器本地时间填写到 X。

## 非目标

- 第一版不支持回复中的图片、视频、GIF、投票或线程。
- 不实现插件本地定时发送作为自动降级方案。
- 不绕过 X 的登录、验证码、风控或账号权限。
- 不在 X 不支持原生回复排期时改成立即发送。

## 已选方案

采用 X 原生回复排期，并设置能力检测门槛。插件只有在回复编辑器中识别到原生排期入口后才继续；缺少入口时停止队列并明确报错。

未采用的方案：

- 插件本地定时：要求浏览器在发送时保持运行，语义不是 X 原生排期。
- 原生与本地混合：同一队列行为不一致，增加误发送和恢复复杂度。

## 用户界面

### 独立入口

排期窗口顶部提供两个入口：

- 原创排期：保留现有原创帖工作流。
- 回复排期：新增批量回复工作流。

两个入口共享全局排期规则和目标时区，但各自保存草稿、队列与预览状态，互不混合。

### 目标时区选择器

目标时区为全局设置，同时影响原创帖和回复：

- 支持按城市、英文时区名或 UTC 偏移搜索。
- 主标签显示当前排期日期对应的 `UTC 偏移 + 城市`，例如 `UTC−07:00 · 旧金山`。
- 辅助文本显示 IANA 时区名，例如 `America/Los_Angeles`。
- 内部只保存 IANA 时区名，不保存固定 UTC 偏移。
- UTC 偏移按每条排期日期动态计算，因此夏令时变化会自动反映。
- 默认值为 `Asia/Shanghai`，显示为 `UTC+08:00 · 北京`。

排期预览同时显示：

- 目标时区的日期和时间。
- 浏览器本地时区的换算日期和时间。

### 回复批量格式

回复使用独立分隔符，不和原创帖队列混排：

```md
timezone: America/Los_Angeles

--- reply ---
id: reply-001
url: https://x.com/example/status/123

第一条回复内容。

--- reply ---
id: reply-002
url: https://x.com/example/status/456
scheduled_at: 2026-06-19 15:30

第二条回复内容。
```

规则：

- `url` 和回复正文必填。
- `id` 可选，缺省时自动生成。
- `scheduled_at` 可选，并始终按当前目标时区解释。
- 文件级 `timezone` 可选；存在时必须与界面选择一致，否则阻止导入并提示切换时区。
- 只接受指向具体状态的 `x.com` 或 `twitter.com` URL；导入时规范化为 `https://x.com/<user>/status/<id>`。
- 重复目标 URL 显示警告但不自动删除，允许用户有意安排多条回复。

## 时区与排期语义

目标时区是排期规则的时间坐标：

- 每日窗口、手动时间、文档时间、均匀分布、固定间隔和随机波动都先在目标时区中计算。
- 计算结果转换为绝对时间戳保存。
- 执行时再把绝对时间戳转换成浏览器本地日期字段，交给现有 X 原生排期弹窗。
- 原创帖的现有 `timezone` 校验从“必须等于浏览器时区”改为“必须等于已选目标时区”。
- 显式时间落在夏令时跳时造成的不存在或重复本地时间时，阻止排期并要求用户更换时间，不静默偏移。

例如，选择 `America/Los_Angeles` 并设置当地 `09:00`，插件在夏令时期间保存对应绝对时间，并在北京时间浏览器中向 X 填入次日 `00:00`。

## 架构

### Popup

负责：

- 原创/回复入口切换。
- 回复队列解析和验证。
- 目标时区选择、持久化和显示。
- 调用统一排期引擎生成绝对时间戳。
- 双时区预览。
- 启动、停止和恢复回复队列。

排期计算继续保持单一入口。现有 `schedulePosts()` 应演化为可接收目标时区的通用排期模块，原创和回复均复用相同时间计算，不在两个入口复制算法。

### Background service worker

回复处理会跨目标页面导航，页面刷新会销毁 content script 状态，因此回复队列由后台负责编排并持久化。

后台负责：

- 保存队列、当前索引、已完成项目、失败项目和日志到 `chrome.storage.local`。
- 将目标标签页导航到当前回复 URL。
- 等待标签页加载完成，再向 content script 下发单条回复任务。
- 收到成功结果后推进索引并打开下一个目标 URL。
- 收到失败结果后停止，不自动跳过。
- 扩展 service worker 被系统挂起后，根据持久化状态恢复当前运行信息。

原创帖现有执行路径不在本功能中强制迁移，避免无关重构。

### Content script

content script 每次只处理一条回复：

1. 确认当前 URL 与目标状态 ID 一致。
2. 确认目标帖子存在且页面已稳定。
3. 点击目标帖的回复入口。
4. 定位回复编辑器并填写纯文字正文。
5. 回读并核对完整正文。
6. 在回复编辑器范围内检测原生排期按钮。
7. 设置排期弹窗中的浏览器本地日期和时间。
8. 确认排期设置。
9. 点击最终的排期按钮，并确认编辑器已完成提交。
10. 向后台返回成功或结构化错误。

选择器必须限定在当前目标帖子及其回复编辑器范围内，不能误用页面上的原创帖编辑器。

## 运行状态

回复队列状态包含：

- `idle`
- `running`
- `stopping`
- `failed`
- `done`

每个项目包含：

- `id`
- `targetUrl`
- `targetStatusId`
- `text`
- `scheduledEpochMs`
- `targetTimezone`
- `status`
- `error`
- `completedAt`

从失败项恢复时，仅重新执行 `failed` 项及其后的 `queued` 项；状态为 `scheduled` 的项目不得重复执行。

## 错误处理与安全边界

以下情况立即停止整个回复队列：

- 无法打开或识别目标帖子。
- 账号未登录、遇到验证码、风控或访问限制。
- 找不到目标帖对应的回复入口或回复编辑器。
- 回复正文写入不完整或回读不一致。
- 回复编辑器没有原生排期按钮。
- 排期弹窗结构无法识别或时间控件设置失败。
- 最终提交状态无法确认。

安全规则：

- 未通过能力检测时绝不点击立即回复按钮。
- 不把失败任务静默标为完成。
- 不自动跳过失败项继续后续批量操作。
- 日志同时记录项目编号、目标 URL、步骤和原始错误，便于恢复和定位 X 页面改版。

## 验证策略

### 自动测试

采用测试先行方式覆盖：

- `--- reply ---` 批量解析、字段校验和 URL 规范化。
- 文件时区与界面时区不一致时拒绝导入。
- 目标时区墙上时间与绝对时间戳互转。
- `America/Los_Angeles` 夏令时和冬令时的偏移变化。
- 不存在或重复的夏令时本地时间被拒绝。
- 原创和回复复用同一目标时区排期规则。
- 自动排期、手动时间和文档时间均按目标时区解释。
- 回复队列成功推进、失败停止和从失败项恢复。
- 已排期项目不会在恢复时重复执行。

### 扩展检查

- `node tools/validate-extension.mjs`
- `node --check x-native-scheduler-test-plugin/background.js`
- `node --check x-native-scheduler-test-plugin/content.js`
- `node --check x-native-scheduler-test-plugin/popup.js`

### 浏览器实测

使用自有测试帖子和测试账号验证：

1. 单条回复可进入 X 原生定时流程，且不会立即发布。
2. 两条以上、不同目标 URL 的队列可逐条完成。
3. 选择北京和旧金山时，X 弹窗时间与双时区预览一致。
4. 回复框没有排期按钮时，队列在最终提交前安全停止。
5. 人为制造目标 URL 失败后，可从失败项恢复且不重复前序项目。

## 验收标准

- 存在独立的批量回复排期入口。
- 原创和回复共用可搜索的目标时区选择器。
- 时区显示采用动态 UTC 偏移、城市和 IANA 名称。
- 回复预览显示目标时区与浏览器本地时区。
- 每条任务按目标 URL 打开对应帖子并使用回复编辑器。
- 只有识别到 X 原生排期能力时才允许最终提交。
- 成功任务进入 X 的定时列表，不会立即发布。
- 队列失败可恢复，已成功任务不会重复执行。
- 第一版只处理纯文字回复。

## 实施顺序约束

第一项实现工作是建立最小浏览器能力探针，验证当前 X 回复编辑器确实暴露原生排期入口。若能力探针失败，则停止后续自动化实现并报告平台限制；不得在未获用户批准时改为插件本地定时发送。
